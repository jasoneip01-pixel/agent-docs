01第一性原理：为什么需要变更面分析

一个安全版本不保证下一个版本安全。

Agent Skill 是 versioned bundle。v1.8.1 是安全的，v1.8.2 可以只改一行就破坏一切。

问题：你怎么知道一个 Skill 版本更新后，Agent 的攻击面没有扩大？

传统解决方案是每次完整扫描。但完整扫描没有基线，你看到的是"这个 skill 有不安全的东西"，而不是"这个 版本 新增了什么不安全的东西"。

维度静态扫描（Skill Scanner）变更面分析（Surface Diff）
视角单个版本绝对风险版本间相对风险增量
结论"SKILL.md 有注入""v1.8.2 新增了 system_prompt_append，v1.8.1 没有"
PR 集成扫描 → 阻塞diff 基线 → 精确判定增量 → 阻塞有问题的增量
假阳性处理花时间排查已知问题只关心新增问题，已存在的不再报警

推导链

  1. Agent Skill = 可版本化的能力包（manifest + scripts + instructions + tool schemas）
  2. 每个版本更新都可能无意或有意地扩大攻击面
  3. 单版本扫描只能回答"这个 skill 有没有问题"，不能回答"这个版本有没有让事情变糟"
  4. 版本变更检测需要：基线扫描 → 增量 diff → 规则判定
核心认知：Agent 的安全不是一次性的。它是一个边界合约——每次更新都需要检查合约是否被篡改。Skill Surface Diff 做的是合约差异审计：你同意的能力边界是什么，这个版本是否越界了。

为什么这比单次扫描更重要

  • PR 审查者不能读每个 SKILL.md — 但可以看一个结构化的 diff 报告："新增网络权限"、"新增高危工具"、"新增指令注入"
  • 语义版本不等于安全版本 — patch 版本可以偷偷加 data_exfiltration。谁检查？
  • 供应链攻击是在版本间发生的 — 一个被攻陷的 maintainer 账号可以 push 一个 v1.8.2 的新 commit。diff 能抓到这个。

核心原语

🔍 基线扫描

skill-surface-diff scan ./skill/ --save-baseline
为 skill 创建初始能力基线快照，存储到 .skill-surface/baseline/

📊 差异分析

skill-surface-diff diff --baseline v1.0 ./v1.1/
与基线对比，标记所有新增、删除、修改的能力边界变更

⚖️ 风险评分

加权公式：权限新增 × 40 + 注入 × 15 + 脚本变更 × 8 + schema 变更 × 5 + 网络扩展 × 5

🚫 规则阻断

规则优先于风险评分。permission.expansion: block 直接阻断——评分只用于优先级，不做决策

04技术架构

核心原语：Surface Diff Engine
baseline = Surface.extract(skill@v1.8.1)
current = Surface.extract(skill@v1.8.2)
deltas = SurfaceDiff.compare(baseline, current)
if delta adds: prompt_override / data_exfiltration / new_high_risk_tool / network_egress → block merge

Diff 流程

Step 1 → Surface.extract(old/) — 扫描旧版本，提取能力面
Step 2 → Surface.extract(new/) — 扫描新版本
Step 3 → ManifestAnalyzer.diff(old, new) — field + capability 级别差异
Step 3a → ScriptAnalyzer.diff(old, new) — 新增/删除/修改脚本检测
Step 3b → PermissionAnalyzer.diff(old, new) — 网络/文件/工具权限差异
Step 3c → ToolSchemaAnalyzer.diff(old, new) — schema 结构差异
Step 4 → Engine.evaluate(deltas, policy) — 规则判定 + 风险评分
Step 5 → Report.build(deltas, verdict) — 输出

差异化核心

🔄 唯一做版本能力差异

Snyk 和 Skill Scanner 检查一个版本是否危险。我们检查这个版本是否比上一个版本更危险——这是 PR 场景的真实需求。

📐 Tool Schema 结构 diff

工具声明从只读变成读写？参数从 optional 变成 required？新工具加到了 payment 命名空间？Schema diff 是其他工具不做的。

⚙️ 规则优先于评分

permission.expansion: block → 直接阻断。风险评分只用于排优先级，不用于决策。Policy-as-code 才是 gate。

🧩 可组合的基线

基线可缓存、可版本化、可复用。适配 Skill Registry 场景：从 registry 拉基线，本地 diff，一致的行为。

与 Skill Scanner 的关系

互补，不是替代。

skill-scanner scan ./new-skill/ → 这个 skill 是否包含恶意代码？
skill-surface-diff diff v1.8.1/ v1.8.2/ → 这个版本的攻击面是否比上一版扩大了？

两者一起用：单版本安全 + 版本差异审计 = 完整的 Agent Skill 供应链安全。
